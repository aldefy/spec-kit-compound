planverify

Judge the plan before execution. The cheaper, earlier mirror of intentguard — catching scope drift while it is still free to fix, judged by a model with no stake in the plan.

The problem

LLMs are built to please. As context grows they affirm, drift, and quietly expand scope — a "while I'm in here, let me also…" reflex. Spec-driven development adds checkpoints, but the existing guard (intentguard) only judges the git diff after code is written. By then a drifted plan has already become drifted code: expensive to catch, messy to unwind.

planverify moves the checkpoint one step earlier — it judges the proposed plan, before any file is touched.

How planverify works inside

1 · Surface analysis (mechanical)

The orchestrator extracts every file the plan proposes to touch and classifies each against the locked intent: in-scope, out-of-scope, or undeclared. Out-of-scope/undeclared paths become drift candidates — recorded as plain facts, not yet judged.

2 · The firewall

A plan is the planner's own justification. Asking it "is my plan good?" is motivated reasoning. So planverify seals a briefing containing only the locked intent, locked expectations, the plan, the tasks, and the surface facts — never the planner's reasoning — and hands it to an independent checker.

3 · The independence ladder

  • Tier 1 — cross-model: a different-vendor model (Claude → Codex/Gemini). Different weights + fresh context + different priors. Strongest.
  • Tier 2 — cross-tier: a cheaper sibling (Opus → Haiku).
  • Tier 3 — same-model, fresh context: the honest floor.

4 · Verdict

PASS no drift, full coverage   REPLAN_ALLOWED small justified drift or a coverage gap   BLOCKED_DRIFT unjustified scope expansion or constraint contradiction

Two real runs

Validated end-to-end on two open-source Android repos, each with a deliberately drifting plan. Both were judged by GPT-5 Codex — a different vendor that never saw the planning. Both returned BLOCKED_DRIFT.

aldefy/kaizen-android

"mark a task complete from the list"

The repo's setCompleted() already exists, so the feature needs only the tasks/ UI. The plan drifted into a schema migration, a new repo method, reminder cleanup, a feature flag, and a social post.

CheckVerdict
P3a — 6 out-of-scope paths (data/reminders/flags/social)BLOCKED_DRIFT
P3d — C1 (new repo method), C2 (schema migration)BLOCKED_DRIFT
P3d — C3 (route through ViewModel)PASS

The checker distinguished the one in-scope-correct decision (C3) from the drift — it wasn't blanket-rejecting.

aldefy/nowinandroid · Now in Android

"reading history screen"

NiA already tracks viewedNewsResources, so History is a new feature/history module reading existing state. The plan drifted into a proto schema bump, a new repo method, a model field, and a bookmarks-screen tweak.

CheckVerdict
P3a — 4 out-of-scope paths (datastore/data/model/bookmarks)BLOCKED_DRIFT
P3d — C1, C2, C4BLOCKED_DRIFT
P3c — E6 large-history eager loadREPLAN_ALLOWED

The E6 flag is the punchline: the plan loaded the full viewed list into memory — the exact eager-load failure from the live demo (all 311 articles on launch), caught at plan time, before a line of code.

planverify catches scope drift and coverage gaps in the plan — by a model with no stake in it — at the cheapest possible place to catch them.

The gate (opt-in, cross-vendor)

By default planverify is advisory. Set SKC_PLANVERIFY_GATE=block to enforce it. A PreToolUse hook — working under both Claude Code and Codex CLI (the converged exit-2 contract) — then blocks the first source-file write when the latest verdict is missing or BLOCKED_DRIFT; doc and spec writes are never blocked. A spec-kit before_implement hook additionally gates /speckit-implement.
