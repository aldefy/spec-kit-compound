Why this exists

LLMs are built to please. As context grows they affirm, drift, and quietly expand scope — a "while I'm in here, let me also…" reflex that turns a focused change into a sprawling one. Raw prompting has no checkpoints, so there is no moment for a human to intervene before the slop lands.

Spec-driven development (SDD) — which predates AI — fixes this by breaking work down and validating at each step. SpecKit brings SDD to coding agents. spec-kit-compound adds the missing discipline on top of SpecKit: lock the boundary before planning, complete the obligation set, and have a model with no stake in the work judge it — before code (planverify) and after (intentguard).

It's an extension, not a harness. It sits on top of whatever harness SpecKit drives — Claude Code, Codex, Cursor, Gemini CLI — and injects intent / expectations / compound discipline around the standard chain.

ICE — intent-driven scoping

Before any planning, an interview captures three things and locks them:

Intent

What to build — and explicitly what not to touch. In-scope and out-of-scope are declared up front so drift is measurable against a boundary.

Context

The why, how, and where — e.g. a front-end change that depends on a back-end contract. Removes ambiguity before design starts.

Expectation

Positive scenarios and hard boundaries, kept in a separate, builder-hidden file — compartmentation that defends against reward-hacking (the builder can't teach to the test).

Compound engineering memory

A local store (docs/compound/) of ADRs, corrections, and patterns — decisions made, mistakes hit, fixes that worked. It persists across features, so learnings carry forward instead of evaporating when the context window does. It's committed to git, so a teammate (and their LLM) reads the same hard-won context. Each feature ends by writing back what it learned — the system gets smarter every cycle.

planverify and intentguard are the two validation gates — same mechanism, different moments. planverify asks "is the proposed plan drifting, before files are edited?"; intentguard asks "did the actual diff drift, after implementation?" Catching it at plan time is far cheaper than unwinding it from a diff.

The firewall — why a different model judges

Whoever did the work is the worst judge of whether it stayed in scope — they have motivated reasoning and they remember what they meant. So both gates seal a briefing containing only the locked criteria, the artifact under judgment (plan or diff), and mechanical facts — never the builder's reasoning — and hand it to an independent checker that climbs an independence ladder:

  • Tier 1 — cross-model: a different-vendor model (Claude → Codex/Gemini). Different weights + fresh context + different priors. Strongest.
  • Tier 2 — cross-tier: a cheaper sibling (Opus → Haiku).
  • Tier 3 — same-model, fresh context: the honest floor.

planverify returns PASS REPLAN_ALLOWED BLOCKED_DRIFT; intentguard returns PASS / REVIEW / BLOCKED. The verdict records who judged and at what tier — so "the builder and the checker never share a workspace" is literally true, never overstated.

Two real runs (planverify)

Validated end-to-end on two open-source Android repos, each with a deliberately drifting plan. Both were judged by GPT-5 Codex — a different vendor that never saw the planning. Both returned BLOCKED_DRIFT.

aldefy/kaizen-android

"mark a task complete from the list"

The repo's setCompleted() already exists, so the feature needs only the tasks/ UI. The plan drifted into a schema migration, a new repo method, reminder cleanup, a feature flag, and a social post — 6 out-of-scope paths. The checker also PASSed C3 (route through the ViewModel), proving it distinguished the one in-scope-correct decision from the drift.

aldefy/nowinandroid · Now in Android

"reading history screen"

NiA already tracks viewedNewsResources, so History is a new feature/history module reading existing state. The plan drifted into 4 out-of-scope paths — proto schema, a new repo method, a model field, and the bookmarks screen. Bonus: obligation coverage flagged E6 (large history) as REPLAN_ALLOWED because the plan loaded the full list into memory — the exact eager-load failure from the live demo (all 311 articles on launch), caught at plan time, before a line of code.

The system surfaces drift — and coverage gaps — by a model with no stake in the work, at the cheapest possible place to catch them. It doesn't auto-fix; it makes the human intervention land early.

Enforcement (opt-in, cross-vendor)

By default the gates are advisory. Set SKC_PLANVERIFY_GATE=block to enforce planverify: a PreToolUse hook — working under both Claude Code and Codex CLI (the converged exit-2 contract) — blocks the first source-file write when the latest verdict is missing or BLOCKED_DRIFT; doc and spec writes are never blocked. A live localhost dashboard visualizes the whole chain — every stage, the planverify drift panel, file diffs, token spend, task progress — read-only, no dependencies.
